Task Summary

This task replaced setup run/setup apply's bare stdin prompt with huh confirmations (a Confirm defaulting to No, and an Input enforcing the same exact-phrase gate as before) and added a bubbles/v2 spinner for project hydrate and scan — both only when stdin and stdout are real terminals. Every interactive path has a plain, script-safe fallback that is byte-for-byte identical to the pre-existing behavior when piped, so no existing script or test needed to change. secretInput is untouched, as required.

What This Task Proves

  • Real, working vhs recordings show the huh Confirm rendering with Yes/No buttons (No selected by default) and the bubbles spinner animating mid-clone — not just described, actually captured from a live PTY session.
  • A genuine bug was caught and fixed during this work: wrapping huh/bubbletea's own output through styledWriter (the same colorprofile-based wrapper used elsewhere) corrupted the TUI's cursor-control escape sequences, making the interactive UI invisible even though the underlying logic still worked. bubbletea v2 already handles its own color downsampling internally, so interactive programs must receive the raw writer.
  • The non-interactive fallback is byte-for-byte identical to the original confirmSetup prompt and error text, confirmed by both a CLI transcript and the pre-existing TestSetupRunCommandRequiresConfirmation test passing unmodified.
  • Two new tests drive huh's real Confirm/Input validation logic via its accessible mode (a genuine, non-bypassed huh code path), closing the same class of TTY-only-rendering gap the planning audit flagged for tasks 01–02.

Artifact: a real bug caught mid-task — styledWriter corrupts interactive TUI output

What it proves: The first recording attempt showed no visible Confirm UI at all (the frame jumped straight from the typed command to the rejection error), because WithOutput(styledWriter(out)) was double-processing bubbletea's own ANSI/cursor-control stream through colorprofile.Writer, corrupting it.

Why it matters: bubbletea v2 (and huh, built on it) already performs its own color-profile detection and downsampling internally — wrapping its output a second time is not just redundant but actively breaks rendering. This is exactly the kind of defect that manual "it compiles, ship it" verification would have missed; only actually driving the interactive path through a real pty surfaced it.

Fix: internal/devspace/interactive.go — pass out directly to huh.NewForm(...).WithOutput(out) and tea.NewProgram(m, tea.WithOutput(out)), never through styledWriter.

Result summary: After the fix, the Confirm UI and spinner both render correctly, as shown in the artifacts above.
